The authoritative `orchestrator-state.json` contract is enforced by `validate_orchestration_artifacts` (artifact_type `orchestrator-state`), whose validator lives in the bundled npm package `@danmoisan/drm-copilot-mcp` at `resources/scripts/dev_tools/validate_orchestration_artifacts.py` — NOT in the repo's `.claude/schemas/orchestrator-state.schema.json` (that JSON schema is looser and only governs the `remediation_loop` shape). Build the checkpoint to the validator from the start to avoid an end-of-run remediation.

Required keys (all must be present): `objective`, `change_budget_estimate`, `path_selected`, `promotion-type`, `short-name`, `relativeFile`, `long-name`, `issue-num`, `feature-folder`, `work-mode`, `plan-path`, `completed_steps`, `next_step`, `last_updated`, `step5_status`..`step10_status`, `delegation_receipts`, `blocked_reason`.

- `stepN_status` enum (verified 2026-07 against bundled `validate_orchestrator_state.py` `VALID_STEP_STATUS`): `not-applicable | pending | delegated | verified | blocked | not_started | in_progress | completed`. `completed` (with the `d`) and `in_progress` ARE valid; the invalid values that fail are `complete` (no `d`) and `passed`. Use `completed` for finished steps.
- `blocked_reason` enum: `none | spawn_agent_unavailable | delegation_launch_failed | delegate_no_receipt | delegate_contract_incomplete | validator_failed | user_requested_stop` (or null).
- `delegation_receipts` MUST be a **list** of objects, each with keys: `step`, `agent_name`, `agent_id`, `skill_source`, `started_at`, `completed_at`, `result_signal`, `artifact_paths` (a list). This contradicts the orchestrator prompt, which describes `delegation_receipts.promotion.{potential_entry,issue,feature_folder}` as a nested object — that object shape FAILS the validator. Put promotion data under a different key (e.g. `promotion_receipts`).
- `work-mode` is hyphenated and must be `minor-audit | full-feature | full-bug` (normalize legacy `full` to `full-feature`).
- `require_complete=true` additionally: rejects any step status of `pending`/`blocked`; requires `blocked_reason` null/`none`; requires `pr_gate` `{pr_number, pr_url, head_branch, head_sha}` and `ci_gate` `{conclusion:"success", head_sha, verified_at}` with `ci_gate.head_sha == pr_gate.head_sha`; and runs the routing-contract check (`_orchestrator_state_routing.py`) which requires top-level `required_agents`/`required_skills`/`required_mcp_tools` arrays that EXACTLY equal (order-sensitive) the selected route's config lists, one `delegation_receipts[].agent_name` per required agent, one valid `skill_receipts[]` per required skill, one `mcp_call_receipts[]` (ok:true) per required MCP tool, and `local_execution_overrides`/`delegation_bypasses` present as EMPTY lists.
- The PR-creation hook (`enforce-pr-author-skill.ps1`) runs a lighter gate (`--require-pr-creation-ready`): when `scripts/dev_tools` is absent it falls back to the portable `Test-OrchestratorStatePrCreationReadiness` in `.claude/lib/orchestrator-state/OrchestratorState.psm1` — base presence + steps 5-8 not `pending`/`blocked` + empty override lists. It does NOT require `pr_gate`/`ci_gate` (those cannot exist pre-PR).
- Skill receipts at `require_complete` (verified 2026-07-07 on #120/PR #123): a `skill_receipts[]` entry is required for EVERY `required_skills` name — including `orchestrator-workflow` and `repo-automation-adapter`, which have NO `SKILL.md` in the repo (record them with a truthful "exercised: ..." evidence string, not a file read). `require_model_routing=true` additionally needs a `model_routing_receipts[]` + `complexity_assessments[]` entry per delegated agent/phase (incl. `pr-author`, `commit-message`, `human-exception-runbook`).
- Each `delegation_receipts[]` entry also needs a `phase` key (verified F20/#124). For the `large` route the validator expects receipts for all six required agents (`task-researcher`, `prd-feature`, `atomic-planner`, `atomic-executor`, `feature-review`, `pr-author`) even when some were fulfilled inline (record them with an honest `method:"inline"` note and `ok:true`). `require_model_routing` cross-checks: every `delegation_receipts[].agent_name` needs a `model_routing_receipts[].agent`, and every routing-receipt `phase` needs a matching `complexity_assessments[].phase` — keep agent+phase names consistent across all three arrays.
- Epic-worktree PR-creation quirks (verified F20/#124): under `epic_mode:true` the portable preflight also requires `epic_context.integration_branch` to match the `--base` value; and `collect_pr_context` (MCP) writes `artifacts/pr_context.summary.txt` to the PRIMARY checkout, not the worktree — copy it into the worktree `artifacts/` so the hook (cwd = worktree) finds it, with the `pr_body_<N>.receipt.json` `created_at` strictly newer than that file's mtime.

**Why:** During issue #72 and again F15/#119 the DONE checkpoint first failed validation because step statuses used `complete`/`passed` and `delegation_receipts` was a nested object per the prompt; the validator requires the enum values above and a receipt list. The `large` route also requires the three route arrays + empty override lists.

**How to apply:** Author the checkpoint with these keys/enums on the first write of each step; keep `delegation_receipts` as an append-only list of per-delegation receipts. Run `validate_orchestration_artifacts` (orchestrator-state, `require_complete=true`) before declaring DONE. Related harness context: [[harness-governance]].
