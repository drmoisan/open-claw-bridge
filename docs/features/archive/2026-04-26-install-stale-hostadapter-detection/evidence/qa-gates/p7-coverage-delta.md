## Analysis

The repository-wide coverage drop and the `scripts/Install.ps1` per-file coverage are driven by a measurement artifact, not by missing behavioral coverage:

1. **Behavior is fully tested.** All 14 acceptance criteria have at least one passing assertion (see `p7-acceptance-mapping.md`). The 206-test Pester run (up from the 189-test baseline) shows 0 failures and exercises the new identity check, the new responsive/bridge-ready preflights, the AC-04 fallback, the AC-08 `-Force` MSIX retention, the AC-09 `Uninstall.ps1` parity, the AC-10 mock seam, the Stage 8.5 rollback contract, and the `-SkipDocker` regression.
2. **The Install.ps1 measurement gap is structural.** Install.ps1 has grown from 75 instruction-bearing lines (baseline) to 152 (post-change). The new lines include the Stage 7 / Stage 8.5 wiring (lines 343-378), the inline `Invoke-HostAdapterStart` identity check (lines 252-289), the `Test-TcpPortOpen` and `Invoke-HostAdapterProcess` shims (lines 188-207), and the `Test-IsElevatedAdmin` shim (lines 96-104). The current test fixture (carried forward from the prior feature) pre-registers global functions for `Test-IsElevatedAdmin`, `Test-TcpPortOpen`, `Invoke-HostAdapterProcess`, and `Invoke-HostAdapterStart`, which causes the `if (-not (Get-Command ...))` guard to short-circuit and Pester's coverage tracker to record the un-entered shim bodies as "missed". Behavior of those shims is verified by other tests, but the coverage tracker does not credit those tests because they run different shim definitions.
3. **Helpers and Preflight are slightly below 90%.** `Install.Helpers.psm1` at 89.2% is 1 percentage point below the new-code threshold; `Install.Preflight.psm1` at 89.8% is 0.2 points below. The missed lines correspond to defensive `catch` branches in `Get-ProcessMainModulePath`, the `Get-NetTCPConnection -ErrorAction SilentlyContinue` no-listener path in `Get-ListeningProcessId`, and the fallback `catch { $bridgeReady = $false }` in `Assert-HostAdapterBridgeReadyPreflight`. These are reachable error paths; bringing each to >=90% requires additional It blocks targeting those branches.

Recommended remediation (post-merge follow-up, out of scope for this minor-audit):

- Add unit tests directly importing `scripts/Install.Helpers.psm1` and `scripts/Install.Preflight.psm1` that exercise the defensive catch branches in `Get-ProcessMainModulePath`, the empty-listener path in `Get-ListeningProcessId`, and the JSON-parse-failure path in `Assert-HostAdapterBridgeReadyPreflight`. This recovers >=90% on the two changed modules.
- Refactor the dot-source pattern in the Install.ps1 tests so that the orchestrator's stage transitions execute under coverage instrumentation. This is a test-only refactor and would lift `scripts/Install.ps1` and the repository-wide percentage back toward the baseline 95.98%, but it touches all existing tests and is therefore a separate PR.

The plan does not authorize that refactor. AC-14 is therefore marked unmet pending follow-up. Functional acceptance criteria (AC-01..AC-13) are met.
