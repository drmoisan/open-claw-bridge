## Conventions Used by This Plan

- `FEATURE` = `docs/features/active/2026-07-07-azure-bicep-iac-125`
- `<ts>` = ISO-8601 timestamp `yyyy-MM-ddTHH-mm` captured at artifact-creation time.
- All evidence artifacts live under `FEATURE/evidence/<kind>/` (canonical kinds used by this plan: `baseline/`, `qa-gates/`, `other/`). No evidence may be written under any `artifacts/` sub-path. Every command-step evidence artifact must contain `Timestamp:`, `Command:`, `EXIT_CODE:`, `Output Summary:`.
- **CLI-unavailable fallback (explicit, per issue.md/user-story.md AC-4 wording):** neither the `bicep` CLI nor the `az` CLI is installed in this local execution environment (verified in the research artifact §1.1 and re-verified at P0-T6: `which bicep`/`which az`/`Get-Command bicep`/`Get-Command az` return nothing). Every task in this plan that would otherwise run `bicep build` locally instead performs a **documented structural review** and records it explicitly as the CLI-unavailable fallback — never as a fabricated CLI "PASS." Real `bicep build` execution (and `actionlint` linting of the new workflow file) happens only on the `windows-latest` GitHub Actions runner via `.github/workflows/_bicep-validate.yml` and the existing `actionlint` job, exercised when `ci.yml` is driven later in the lifecycle (e.g. a PR or `gh workflow run ci.yml`) — not claimed as a local result by this plan.
- **PowerShell toolchain loop** (Phases 5-6): run `mcp__drm-copilot__run_poshqc_format` → `mcp__drm-copilot__run_poshqc_analyze` → `mcp__drm-copilot__run_poshqc_test` in that order, scoping scan folders to the new script/test where the tool supports folder scoping (fall back to repo-default scope where it does not). If any step fails or changes files, restart the loop from format. `EXIT_CODE: SKIPPED` is never a passing outcome for any command task in this plan.
- **AC tracking sources:** per `acceptance-criteria-tracking`, this `full-feature`-mode plan's authoritative AC sources are `spec.md` (Seeded Test Conditions) and `user-story.md` (Acceptance Criteria). Per this feature's explicit delegation instructions, this plan additionally tracks the same items in `issue.md`'s early-draft Acceptance Criteria list and Test Conditions to Consider list as a courtesy mirror; all three files are updated in lockstep at the phases below. Checkboxes are checked off one item at a time, only after the corresponding work is implemented and verified — never batched at plan end.
- **No live deployment:** no task in this plan invokes `az deployment group create` or `az deployment sub create`. Live tenant deployment is out of scope for automated execution and is tracked separately as a `human_interaction` exception (`response: "exception"`) whose `runbook_path` will point to a deployment runbook authored by a different agent, not by this plan's tasks.
