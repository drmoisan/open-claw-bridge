- **PowerShell coverage tooling note (established workaround):** `mcp__drm-copilot__run_poshqc_test` in coverage mode fails on every invocation in this repository because its bundled `pester.runsettings.psd1` hardcodes `CodeCoverage.Path` to files that exist only in the `drm-copilot` source repository (reproduced defect: `#111`, `#125`, `#135`, `#137`, `#139`, `#142`, `#144`). The numeric-coverage source for both baseline and final-QC PowerShell tasks is: import the bundled `PoshQC.psd1` module directly and call `Invoke-PoshQCTest -Root <repo> -SettingsPath <corrected>`, where `<corrected>` is a SCRATCHPAD-only copy of the bundled runsettings with `CodeCoverage.Path` rewritten to this repository's production PowerShell files under `scripts/**` (no production `ExcludedPath` entry per the Coverage Exclusion Policy). Record both the failing MCP invocation and the corrected-runsettings invocation in the evidence artifact.
- **New module function seam (`OpenClawContainerValidation.psm1`):** `ConvertFrom-OpenClawImageReference -ImageReference <string>` -> `[pscustomobject]@{ Repository; Tag }` (splits on the last `:`; returns `Tag = ''`, not a throw, when no `:` is present). `Test-OpenClawImageVersionAligned -CoreImageReference <string> -AgentImageReference <string> -ExpectedVersion <string>` -> a `Get-OpenClawValidationResult`-shaped object (`Category 'Container'`, `Name 'ImageVersionAlignment'`); `IsExpected` is true only when both parsed tags equal `-ExpectedVersion` by exact string comparison AND both independently validate as well-formed 4-part versions via `[System.Version]::TryParse`. Both functions are pure (no I/O, no docker/HTTP seam).
- **New `Install.ps1`-local helper seam (script-scope only, not exported):** `Get-ComposeServiceImageTag -ComposeContent <string[]> -ImageRepository <string>` -> `[string]`, scanning for `^\s{4}image:\s*<ImageRepository>:(.+)$` and throwing a distinct "no matching image: line found" error when absent. `Assert-ComposeImageVersionAligned -ComposeFilePath <string> -ResolvedVersion <string>` reads the compose file once via `Get-Content -LiteralPath`, calls the helper twice (`openclaw/core`, `openclaw/agent`), and throws one aggregated error naming both raw image references, both extracted tags, `$ResolvedVersion`, and `$ComposeFilePath` when either tag is absent or does not equal `$ResolvedVersion`. This is an intentional, small duplication of the module's comparison logic (per the spec's ratified design decision, AC11) rather than an `Import-Module` of `OpenClawContainerValidation.psm1`.
- **Version format:** 4-part, `^\d+\.\d+\.\d+\.\d+$`, matching `Get-ManifestVersion`'s existing validation. Comparison is exact string equality of the extracted tag against `$ResolvedVersion`/`-ExpectedVersion`, not `[System.Version]` numeric equality; `[System.Version]::TryParse` is used only as a secondary, message-quality check to distinguish "malformed tag" from "version mismatch".
- **Fixture note:** neither new production function requires the shared fixture module `tests/scripts/fixtures/OpenClawContainerValidation.Fixtures.psm1` or its `-Global` `Import-Module` pattern, because both are pure and unit-tested via direct `.psd1` import (matching the existing `OpenClawContainerValidation.EnvFilePathDefault.Tests.ps1` pattern, which needs no docker/HTTP seam).

## Open Questions / Notes

- **Line-count risk on `OpenClawContainerValidation.psm1` (452 -> ~507-517 projected):** this is the primary implementation risk identified during planning. The objective's scope is "the OpenClawContainerValidation module" (the module directory), not strictly the single `.psm1` file, and the module directory itself already documents a prior split performed to honor this same cap. A three-rung, in-scope mitigation ladder therefore applies at P1-T5, in order: (1) maximally compact implementation of the two new functions (already specified in P1-T1/P1-T2); (2) condensing the module's existing multi-line comment-based help blocks on the probe functions to single-line `.SYNOPSIS` (behavior-preserving, reclaims an estimated 20-40 lines), which keeps both new functions in `OpenClawContainerValidation.psm1` per the spec's ratified design; (3) only if rungs 1-2 are insufficient, extracting the two new pure functions into a sibling file `OpenClawImageVersionAlignment.psm1` within the same `OpenClawContainerValidation/` module directory, loaded via `OpenClawContainerValidation.psd1`'s `NestedModules` and re-exported through `FunctionsToExport` - this remains within "the OpenClawContainerValidation module" scope and introduces no new `Import-Module` reference from `Install.ps1` (AC11 is unaffected). P1-T5 verifies the ladder's outcome in sequence and reaches a blocked state only if all three rungs fail, which is not expected given the projected overage.
- **No sibling test-file split for `Install.DockerStage.Tests.ps1`:** the file is currently 202 lines (well under the 500-line cap), so unlike prior #142/#144 line-cap mitigations, this plan extends it directly rather than creating `tests/scripts/Install.ImageVersionAlignment.Tests.ps1`. P2-T6 verifies this projection holds after implementation; if it does not, the AC8-named sibling file is the fallback, consistent with the spec's own conditional language.
- **Mismatch mechanism (b) is documented, not re-implemented:** per research Section 4(b)/9 and spec Technical specifications, a post-`docker load` `docker image inspect` cross-check is not added; this is already substantially mitigated by `pull_policy: never` (established by #142) plus `Invoke-ComposeUp`'s existing throw-on-nonzero-exit behavior. No task in this plan re-implements it, and no acceptance criterion requires it.
- **Regression scope is full-suite, not a scoped subset:** per `spec.md`'s Test Strategy, the seven regression files named in AC14 do not have their production targets changed by this fix, but they exercise shared harnesses/fixtures (notably `tests/scripts/fixtures/OpenClawContainerValidation.Fixtures.psm1`) that the new test files also depend on, so P3-T4 runs the full named set rather than a scoped subset.
- **AC11 duplication is intentional:** the `Install.ps1`-local `Assert-ComposeImageVersionAligned`/`Get-ComposeServiceImageTag` pair duplicates the module's pure comparison logic rather than importing `OpenClawContainerValidation.psm1`, because that module is not part of the bundle-staged file set (`Copy-InstallScriptsIntoBundle`). This is a ratified design decision from `spec.md`, not an oversight; P3-T1 and P4-T5 both verify no import was (re-)introduced.
