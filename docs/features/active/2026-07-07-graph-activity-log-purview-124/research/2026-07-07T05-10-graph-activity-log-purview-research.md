## 1. Current State — F9 Audit Seam

- `src/OpenClaw.Core/Agent/Contracts/IActionAuditLog.cs` — two methods only: `Task RecordAsync(ActionAuditRecord record, CancellationToken ct)` and `Task<IReadOnlyList<ActionAuditRecord>> GetByMessageIdAsync(string messageId, CancellationToken ct)`. No overloads, no versioning hooks. Any additive change must be a new field on `ActionAuditRecord`, not a signature change.
- `src/OpenClaw.Core/Agent/Contracts/ActionAuditRecord.cs` — a sealed positional C# record with 13 required-by-position parameters: `Mailbox, MessageId, EventId, ActionType, ActingFlags, CorrelationId, ResultCode, ErrorDetail, OriginalStartUtc, OriginalEndUtc, NewStartUtc, NewEndUtc, RecordedAtUtc`. All existing production/test call sites (`SchedulingWorker.Audit.cs`, `CoreCacheRepositoryAuditLogTests.cs`) construct it with **named arguments**, so appending new optional (nullable, default-valued) parameters at the end of the positional parameter list is additive and non-breaking for every current caller.
- `ActionType` and `ResultCode` are **plain string constants**, not enums (`SentActionKey.ProposalReply = "proposal-reply"`, `ActionAuditResultCode.Sent = "sent"` etc.). The `ActionAuditResultCode` XML doc explicitly states the const-string design exists so "Stage 2 (F18/F19) can append reschedule codes without a contract or schema change." The same extensibility applies directly to F20: new CloudSync activity types are new `const string` values in a new or existing static class (e.g., a new `CloudSyncActivityType` class paralleling `SentActionKey`), not new interface members.
- `src/OpenClaw.Core/CoreCacheRepository.AuditLog.cs` — `IActionAuditLog` implementation on `CoreCacheRepository` (`internal sealed partial class`). Validates six required non-empty fields via `ThrowIfEmpty`: `Mailbox, MessageId, ActionType, ActingFlags, CorrelationId, ResultCode`. The `audit_log` SQL table (`src/OpenClaw.Core/CoreCacheRepository.Schema.cs` lines 107-123) declares `mailbox TEXT NOT NULL` and `message_id TEXT NOT NULL`.
  - **Finding (blocking ambiguity for the planner):** CloudSync activity events do not naturally have a `MessageId` — subscription lifecycle and delta-reconciliation events describe a mailbox-level operation, not a single message. `ActingFlags` is also send/calendar-specific (`SendEnabled=...;CalendarWriteEnabled=...`) and has no CloudSync analogue. The plan must resolve how new CloudSync event types satisfy these two required-non-empty fields: options include (a) a documented sentinel value (e.g., `MessageId = "(cloudsync)"`, `ActingFlags = "N/A"`), (b) relaxing `ThrowIfEmpty`/the SQL `NOT NULL` constraint specifically to permit these two fields to be null/empty for CloudSync event types (a schema-level change but not an interface-level one), or (c) mapping a natural analogue (e.g., subscription id or Graph resource id into `MessageId`). This is not resolvable from evidence alone; it is a design decision that changes either validation logic or schema DDL and should be made explicit in the plan.
- `src/OpenClaw.Core/Agent/Runtime/SchedulingWorker.Audit.cs` — the sole existing consumer. Uses `BuildAuditRecord` (private) and `WriteAuditSafelyAsync` (the "one sanctioned resilience boundary," spec D4: catches all non-cancellation exceptions from `actionAuditLog.RecordAsync`, logs Error, continues). Confirmed this file requires **zero changes** — it does not enumerate `ActionAuditRecord`'s fields exhaustively in any switch/pattern that would break on new optional fields.

## 3. Purview / Graph Activity Log Record Shape (external convention, no live endpoint)

No live Microsoft Purview or Graph activity-log endpoint exists in this environment or in CI; this section is for defining a plausible target record shape for a pure projection only, verified against Microsoft Learn documentation (`https://learn.microsoft.com/en-us/graph/api/resources/directoryaudit`, `directoryAudit` resource, v1.0, captured 2026-07-07):

- `id` (String, GUID) — unique activity id.
- `activityDateTime` (DateTimeOffset, UTC) — when the activity occurred → maps from `ActionAuditRecord.RecordedAtUtc`.
- `activityDisplayName` (String) — human-readable operation name (e.g., "Create subscription", "Renew subscription") → derived from the new `ActionType` constant.
- `category` (String) — resource category grouping (e.g., `UserManagement`, `GroupManagement` in the real API) → would map to a new fixed category such as `CloudSyncActivity` for this feature's records.
- `correlationId` (Guid) → maps directly from `ActionAuditRecord.CorrelationId`.
- `operationType` (String) — `Add`/`Update`/`Delete`/etc. → derivable from the CloudSync event type (create/renew/expire map naturally to Add/Update/Delete-like semantics).
- `result` (String enum: `success`/`failure`/`timeout`) → derived from `ActionAuditRecord.ResultCode`.
- `resultReason` (String) → maps from `ActionAuditRecord.ErrorDetail` when present.
- `initiatedBy` (object: user or app) → for this service, always an app-only identity; a minimal `{ app: { appId, displayName } }` shape or similar is plausible, though there is no existing field on `ActionAuditRecord` carrying an app id — the projection would need to supply a fixed/config-sourced value, not derive one from the record.
- `targetResources` (collection) → would carry the mailbox/subscription id; maps from `ActionAuditRecord.Mailbox` (and, once resolved per §1's open question, whatever field carries the CloudSync subscription/resource id).
- `additionalDetails` (key-value collection) — open-ended extension bag; a natural place to carry fields that don't fit the fixed shape (e.g., the original CloudSync event type string) without widening the fixed schema.

The separate Microsoft Purview unified audit log (a different, compliance-oriented log from the Entra `directoryAudit`/`auditLogs` resource above) conventionally uses field names such as `RecordType`, `Operation`, `Workload`, `UserId`, `CreationTime`, `ResultStatus` — the spec's own wording ("Graph `auditLogs` / Purview unified audit log conventions") acknowledges both are informing conventions, not a single fixed target schema. **The plan should pick one concrete target field set (recommend the `directoryAudit`-style shape above, since it is the one with a stable, documented Graph resource type) and state explicitly that the mapping is aspirational/illustrative, not validated against a live endpoint.**

## 4. Correlation-ID Threading — Existing Mechanism

- `GraphRequestExecutor.ExecuteAsync(..., string? requestId, ct)` (`src/OpenClaw.Core/CloudGraph/GraphRequestExecutor.cs` lines 53-65): "Caller-supplied correlation id; blank generates a GUID." The resolved id becomes `ApiMeta.RequestId` on the returned `ApiEnvelope<T>` (`src/OpenClaw.HostAdapter.Contracts/ApiMeta.cs`: `public sealed record ApiMeta(string RequestId, string AdapterVersion, BridgeStatusDto? Bridge)`), and is also sent as the Graph `client-request-id` header. This is the existing mechanism behind `CreateAsync`/`RenewAsync`/`GraphDeltaReconciler.RunAsync`'s Graph HTTP calls.
- `GraphDeltaReconciler.RunAsync` generates its own `requestId = Guid.NewGuid().ToString()` once per invocation and threads it through every page's Graph call and into `RecordRunAsync`'s `ingest_runs.request_id` column — this is the closest existing precedent for "one GUID per operation, reused across all sub-steps," matching the F9 correlation-id semantics ("worker-generated GUID string for this... evaluation").
- **No existing correlation id flows through webhook receipt** (`NotificationRequestProcessor`) because that path makes no Graph call by design (D-1). A new generation point is required there; recommend `Guid.NewGuid().ToString()` generated once per notification item processed, consistent with the existing inline pattern in `GraphDeltaReconciler` (no `Random.Shared` involved, so no banned-API conflict per `.claude/rules/csharp.md`).
- Conclusion: correlation-id threading for subscription lifecycle and delta-reconciliation events can reuse the existing `ApiMeta.RequestId` / `GraphDeltaReconciler` requestId mechanism; webhook-received events need a new, but pattern-consistent, generation point.

## Rejected Alternatives

- **Alternative: put the Purview projection in a new top-level `Purview/` or `CloudSync/` namespace instead of `Agent/Contracts/`.** Rejected in favor of colocating with `ActionAuditRecord.cs` because the projection's sole input is `ActionAuditRecord` and it has no CloudSync-specific behavior of its own (it is a pure mapping over the already-extended audit shape); colocating keeps the "record + its projections" contract surface in one place, consistent with the repo's existing pattern of grouping `ActionAuditRecord`/`ActionAuditResultCode`/`IActionAuditLog` together in `Contracts/`.
- **Alternative: emit CloudSync audit events into the existing `ingest_runs` table instead of extending `audit_log`.** Rejected because `ingest_runs` already exists for a different purpose (poller/reconciler diagnostic run history, not a per-action audit trail with correlation-id joins across the send and CloudSync domains) and the issue explicitly directs extension of the F9 `audit_log`/`ActionAuditRecord` seam; reusing `ingest_runs` would not satisfy the "joinable across outbound-send and CloudSync activity domains" requirement since `ingest_runs` has no `correlation_id` semantics tying it to `audit_log`'s `CorrelationId` field, only its own independent `request_id`.

## Open Ambiguities for the Planner

1. **Required-non-empty-field gap (§1):** `MessageId` and `ActingFlags` are `NOT NULL`/non-empty-enforced on `audit_log`/`ActionAuditRecord` but have no natural value for CloudSync activity events. A concrete resolution (sentinel value, relaxed validation for new event types, or field repurposing) must be chosen before implementation.
2. **Webhook correlation-id generation point (§2, §4):** no existing id flows through `NotificationRequestProcessor`; a new generation point must be added, and its exact placement (once per request body vs. once per notification item within a batch) should be decided.
3. **Dropped/invalid webhook audit scope (§2):** whether rejected notifications (unknown subscription, mismatched clientState) also generate audit records is unresolved from the spec text and affects the Purview projection's completeness.
4. **Target Purview/Graph field set (§3):** the spec cites both Graph `auditLogs`/`directoryAudit` and the separate Purview unified audit log conventions; a single concrete target shape should be pinned down (this research recommends the `directoryAudit`-style shape as the more concretely documented option) and marked explicitly as illustrative pending live-tenant verification (HI exception + runbook, per the F17/F11 precedent already named in the spec).
