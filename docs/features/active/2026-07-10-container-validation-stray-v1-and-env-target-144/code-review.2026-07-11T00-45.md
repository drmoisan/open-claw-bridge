## Executive Summary

The branch fixes two false-negative failures in `Invoke-OpenClawContainerPathValidation.ps1` against a healthy install: (1) the in-container HostAdapter probe requested `http://host.docker.internal:4319/v1/status`, but HostAdapter serves `/status` at the route root (independently confirmed via `src/OpenClaw.HostAdapter/Program.cs:73-74`, `app.MapGet("/status", ...)`, with no `/v1` route group anywhere in the service) — corrected to drop `/v1`; (2) the gateway-token presence check defaulted `-EnvFilePath` to the repo-root `./.env` template instead of the deployed operator `.env`, corrected via two new pure helpers (`Get-OpenClawOperatorEnvFilePath`, `Resolve-OpenClawDefaultEnvFilePath`) with a documented present/absent fallback. The branch also adds an in-container `OPENCLAW_GATEWAY_TOKEN` check (`Test-OpenClawGatewayTokenInContainer`, routed through the existing docker seam) and corrects dashboard-authentication documentation in `README.md` and `docs/mailbridge-runbook.md` that previously overstated tokenless/auto access. Design is disciplined: both new pure helpers are genuinely I/O-free, the new in-container probe reuses the existing `Invoke-OpenClawDockerCommand` seam, and no HostAdapter source or docker-compose/Dockerfile file is touched. One finding is Blocking: two new tests pass only via the specific MCP test-runner wrapper the executor's own evidence relied on, and fail deterministically under a standard `Invoke-Pester` invocation — a real, root-caused, and independently-fix-verified test defect, not a flaky/environment artifact.

## Design Principles (general-code-change.md)

- **Simplicity first**: both new helpers are small, single-purpose, and free of side effects; the fix to the probe URL is a one-line change plus its accompanying text, with no unrelated refactor.
- **Reusability**: `Test-OpenClawGatewayTokenInContainer` correctly reuses the existing `Invoke-OpenClawDockerCommand` seam and the shared `Get-OpenClawValidationResult` result-shaping helper rather than introducing a parallel path.
- **Extensibility**: `Get-OpenClawOperatorEnvFilePath -LocalAppDataPath` and `Resolve-OpenClawDefaultEnvFilePath -OperatorEnvFilePath -FallbackEnvFilePath` both use named, defaultable parameters (the former defaults to `$env:LOCALAPPDATA` but is fully overridable for testing, matching the repo's adapter-seam convention for environment dependencies).
- **Separation of concerns**: pure path composition/resolution logic is fully separated from the one `Test-Path` filesystem probe, itself confined to `Resolve-OpenClawDefaultEnvFilePath`.
- **Non-goals honored** (all independently confirmed via `git diff --name-only`): zero `src/OpenClaw.HostAdapter/**` paths in the change set; no tracked docker-compose files, Dockerfiles, or `Install.Helpers.psm1` changes; no WebSocket/`ws://`/`wss://`/device-pairing-handshake code added (the only match for pairing-related terms is descriptive help-comment prose stating the absence of a handshake).

## Test Quality

- One behavior per `It`, clear `Describe`/`Context` naming traceable to the ACs, Arrange/Act/Assert structure throughout.
- Hermeticity is generally strong: no temp files, no real `docker`/network calls, mocks target the established seams (`Invoke-OpenClawDockerCommand`, `Invoke-WebRequest`, `Test-Path`, `Get-Content`) with `-ModuleName`-scoped mocking used consistently — **except** the two `It` blocks identified in the Blocking finding, which use an unscoped `Mock` against a function that lives in a different module than the script under test, exposing a latent nested-import defect in the shared fixture.
- Negative paths are present for the new probes (non-200, non-zero docker-exec exit for `HostAdapterInContainer`; absent-token and docker-exec-failure paths for `GatewayTokenInContainer`).
- Determinism: no `Start-Sleep`/timing hacks; all seams are mocked; the one ambient dependency (`$env:LOCALAPPDATA` default parameter on `Get-OpenClawOperatorEnvFilePath`) is fully overridable and is overridden in every test that exercises it.
- The full-suite aggregation-count assertions (`SupportingDiagnostics.Count`) were updated consistently across all three affected pre-existing full-run tests, including one not explicitly enumerated in the plan — correctly generalized rather than causing a missed update.
