# `admin-access-automation` — User Story

- Issue: #148
- Owner: drmoisan
- Status: Draft
- Last Updated: 2026-07-11T20-30
- Epic: openclaw-runtime-remediation (child C, wave 1); depends on child B (installer-image-version-alignment)

## Story Statement

- As an OpenClaw operator, I want the gateway token delivered to the Control UI via a
  ready-to-open `#token=` URL, so that I can authenticate the Control UI without
  manually locating and pasting the token.
- As an OpenClaw operator, I want an automated, idempotent path to rotate or reissue
  the HostAdapter device token, so that I can replace a compromised or aged device
  token and keep both ends in agreement without following an ad-hoc manual procedure.
- As an OpenClaw operator, I want the `web_search` provider provisioned in the agent
  configuration, so that the `web_search` tool has a working provider entry once I
  supply the provider API key.
- As an OpenClaw operator, I want a committed runbook for the steps that require a
  human-held secret, so that the manual parts of admin-access setup are documented and
  repeatable rather than undocumented blockers.

## Problem / Why

Provisioning gateway admin access after an OpenClaw install currently requires
undocumented manual steps. The 2026-07-10 runtime diagnosis found that an operator
must hand-deliver the gateway token to the Control UI, has no supported path to
rotate or reissue the device token, and must configure the `web_search` provider by
hand. These gaps break a clean, repeatable end-to-end admin-access flow and leave
human-held-secret steps uncaptured.

Token generation itself is already solved: `scripts/Invoke-OpenClawAgentOnboarding.ps1`
generates the gateway token (`OPENCLAW_GATEWAY_TOKEN`) into the target `.env`. This
feature provisions delivery, rotation, and provider setup only; it does not generate
tokens.

## Personas & Scenarios

- Persona: OpenClaw operator (deployment/admin role)
  - Who: an engineer or administrator who installs and maintains the OpenClaw runtime
    on a Windows host with Docker.
  - What they care about: a repeatable, documented admin-access flow; not leaking
    secrets; being able to rotate credentials.
  - Constraints: some secrets are human-held (search-provider key, Anthropic key,
    initial device-token secret) and cannot be generated by automation; the Control UI
    authentication happens in a browser.
  - Goals and frustrations: wants to finish admin-access setup without reverse-
    engineering undocumented steps; frustrated by manual token handling and the lack
    of a device-token rotation path.
  - Context and motivations: operating against child B's version-aligned staging with
    matched Control UI and gateway images.

- Scenario: gateway-token delivery
  - The operator has just run onboarding, so `OPENCLAW_GATEWAY_TOKEN` is present in
    `.env`. They run the delivery command, which reads the token and port and returns
    `http://127.0.0.1:18789/#token=<token>`. The operator opens the URL in a browser
    to complete Control UI authentication (the browser step is the documented manual
    handoff). If the token is missing, the command fails with a message pointing to
    the onboarding script rather than producing a broken URL.

- Scenario: device-token rotation
  - The operator needs to replace the HostAdapter device token. They run the rotation
    command, which (after `ShouldProcess` confirmation) writes a new random secret to
    the host token file and restarts `openclaw-core` and `openclaw-agent` through the
    docker seam so both ends read the new value. Where HostAdapter runs interactively,
    the operator restarts it per the runbook. Re-running without a force flag does not
    rotate an already-valid token.

- Scenario: `web_search` provider provisioning
  - The operator provisions the `web_search` provider. Automation adds the provider
    entry and a SecretRef to the baked agent config seed and validates the JSON. The
    operator supplies the provider API key into `.env`/secrets (the human-held step),
    and the image is rebuilt so the seed change persists.

## Acceptance Criteria

### Capability 1 — gateway-token delivery via `#token=`

- [x] US-1.1 Running the delivery command with a present `OPENCLAW_GATEWAY_TOKEN`
  returns the URL `http://127.0.0.1:<OPENCLAW_AGENT_PORT>/#token=<token>`, with the
  port resolved from `OPENCLAW_AGENT_PORT` (default `18789`).
- [x] US-1.2 The base64url token is placed in the fragment unchanged (no re-encoding).
- [x] US-1.3 With a missing or empty token, the command fails with a clear error
  pointing to `Invoke-OpenClawAgentOnboarding.ps1` and emits no URL.
- [x] US-1.4 The token value never appears in command output, verbose, debug, or logs.
- [x] US-1.5 The runbook documents opening the URL in a browser to complete Control UI
  authentication and the post-recreation re-pair (clear browser site data + reopen).

### Capability 2 — device-token rotation/reissue

- [x] US-2.1 Running rotation writes a new cryptographically-generated non-empty secret
  to the host device-token file before restarting consumers.
- [x] US-2.2 Rotation restarts `openclaw-core` and `openclaw-agent` via the
  `Invoke-OpenClawDockerCommand` seam so all consumers read the rotated value.
- [x] US-2.3 Rotation prompts via `ShouldProcess` for the file write and each restart.
- [x] US-2.4 Re-running rotation without an explicit force flag does not rotate an
  already-valid token (idempotent).
- [x] US-2.5 Rotation fails explicitly on an unreadable/unwritable token file or a
  docker restart failure, and directs the operator to the runbook when the host token
  file is absent (no silent placeholder).
- [x] US-2.6 The device-token value never appears in output, verbose, debug, or logs.
- [x] US-2.7 The runbook documents supplying/keeping the initial device-token secret
  and restarting an interactively-run HostAdapter during rotation.

### Capability 3 — `web_search` provider provisioning

- [x] US-3.1 Provisioning adds or validates a `web_search` provider entry in
  `deploy/docker/openclaw-assistant/openclaw.json` referencing the provider API key via
  a SecretRef-style env interpolation, with no hard-coded key.
- [x] US-3.2 Provisioning is idempotent (no duplicate provider entries) and validates
  the resulting JSON, failing explicitly on invalid JSON or a missing referenced key.
- [x] US-3.3 The change is made in the baked seed file; the operator understands the
  image must be rebuilt for the change to persist.
- [x] US-3.4 The runbook documents supplying the search-provider API key (external
  SaaS-issued, human-held) into `.env`/secrets.

### Capability 4 — committed runbook for human-held-secret steps

- [x] US-4.1 A committed runbook enumerates every human-interaction / human-held-secret
  step: browser Control UI authentication; post-recreation site-data clear and reopen;
  search-provider API key; Anthropic key in `secrets/.env.anthropic`; interactive
  HostAdapter restart during rotation; initial host token file provisioning.
- [x] US-4.2 Each runbook step states what the operator supplies or does and where
  automation hands off to the operator.
- [x] US-4.3 The runbook is cross-linked from or integrated with the canonical operator
  runbook so it is discoverable.

## Non-Goals

- Generating any token; `Invoke-OpenClawAgentOnboarding.ps1` remains the sole gateway-
  token generator and is unchanged.
- Generating or issuing human-held secrets (search-provider API key, Anthropic key,
  initial device-token secret).
- Automating the browser-side Control UI authentication and device-pairing handshake.
- Modifying child B's installer files or assuming they are present in this worktree.
