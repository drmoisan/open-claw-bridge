## Acceptance Criteria (from `spec.md`, numbered AC1–AC12)

- AC1 — Image tar produced: `Publish.ps1` invokes `docker build` for both images and a single `docker save` writing `<BundleRoot>/docker/openclaw-images.tar` naming all four refs (`openclaw/core:<version>`, `openclaw/core:pre-mvp`, `openclaw/agent:<version>`, `openclaw/agent:pre-mvp`). Verified by Pester argument-vector assertions against the mocked `Invoke-DockerExe` seam.
- AC2 — Build args mirror compose: core build passes `-f deploy/docker/openclaw-core.Dockerfile --target runtime --build-arg BUILD_CONFIGURATION=<Configuration>`; agent build passes `-f deploy/docker/openclaw-agent.Dockerfile --build-arg OPENCLAW_AGENT_IMAGE=<resolved-from-.env-or-default>`; both use the repo root as context. Verified by exact arg-vector tests plus a code comment cross-referencing `docker-compose.yml`.
- AC3 — Bundle compose transformed: the emitted bundle `docker/docker-compose.yml` contains no `build:` key, contains `pull_policy: never` on both services, references `image: openclaw/core:<version>` and `image: openclaw/agent:<version>`, and preserves every other line of the tracked compose. Verified by pure-function tests on string fixtures.
- AC4 — Transform fails fast on drift: `Convert-ComposeToBundleCompose` throws when the expected `build:`/`image:` keys for either service are absent. Verified by negative-path unit tests.
- AC5 — Dev compose path unaffected: the tracked repo `docker-compose.yml` still contains both `build:` blocks and the `openclaw/*:pre-mvp` image tags (no diff to the tracked file in the change set).
- AC6 — Manifest correct at any size: `manifest.json` includes `docker/openclaw-images.tar` with a correct SHA-256, and `New-ManifestEntry` returns `size` as `[long]`, verified by a unit test using a length greater than 2,147,483,647.
- AC7 — Install loads before compose up: `Install.ps1` Stage 9 calls `Invoke-DockerImageLoad` on `<DestDockerDir>/openclaw-images.tar` before `Invoke-ComposeUp`, and the entire step is skipped under `-SkipDocker`. Verified by stage-sequence tests.
- AC8 — Load failure modes are explicit: `Invoke-DockerImageLoad` throws with remediation text naming the expected tar path when the tar is missing, and throws on non-zero `docker load` exit. Verified by unit tests.
- AC9 — Wrapper seam in place: `scripts/Publish.Docker.psm1` and `scripts/Install.Docker.psm1` exist, each under 500 lines, each defining a module-scoped `Invoke-DockerExe -DockerArgs <string[]>` seam; every docker invocation introduced by this fix routes through the seam; the four pre-existing direct `docker` call sites in `Install.Helpers.psm1` are unmodified.
- AC10 — Bundle staging complete: `Copy-InstallScriptsIntoBundle` stages `Install.Docker.psm1`, and `Copy-DockerArtifact` no longer copies `docker-compose.dev.yml`. Verified by updated `Publish.Helpers.Tests.ps1` cases.
- AC11 — Tests are hermetic: `tests/scripts/Publish.Docker.Tests.ps1` and `tests/scripts/Install.Docker.Tests.ps1` mock only the `Invoke-DockerExe` seam (no real docker, no `global:docker` shim for new code, no temp files) and cover positive, negative, and `-WhatIf` paths.
- AC12 — No regression, clean toolchain: the full existing Pester suite passes with the documented test updates applied, and PoshQC format + analyze report clean on all new and modified PowerShell files, in a single full pass.
