## Current State Summary

The repository already implements a working Local MVP (Stage 0) skeleton across all six layers described in the master document's delivery-stage framing, at a materially more advanced state than the master document's own "Current state and Local MVP work" subsection describes (that subsection is now stale relative to the code — see below).

**Layer 1 — `OpenClaw.MailBridge`** (COM bridge, T2). Six original read RPC methods plus a seventh, `send_mail` (`src/OpenClaw.MailBridge.Contracts/Models/BridgeContracts.cs:20-40`), backed by live Outlook COM send (`src/OpenClaw.MailBridge/OutlookComMailSender.cs`, `src/OpenClaw.MailBridge/SendMailRpcHandler.cs`). `EventDto` and `MessageDto` carry the Graph-shaped fields introduced by issues #71–#73 (`RequiredAttendeesJson`/`OptionalAttendeesJson`/`ResourcesJson`, `Categories`, `IsOrganizer`, `IsOnlineMeeting`, `AllowNewTimeProposals`, `ICalUId`, `SeriesMasterId`, `LastModifiedDateTime`, `BodyFull`, `SensitivityLabel`, `ResponseStatus`, and message-side `SenderEmailResolved`/`FromEmailAddress`/`ConversationId`/`MeetingMessageType`) — `src/OpenClaw.MailBridge.Contracts/Models/BridgeContracts.cs:76-128`. The bridge's own cache schema already has a `response_status` column (`src/OpenClaw.MailBridge/CacheRepository.Schema.cs:43-45`).

**Layer 2 — `OpenClaw.HostAdapter`** (T1). Evolved from the bespoke `/v1/*` surface to a Graph-shaped surface: `GET /users/{id}/mailboxSettings`, `GET /users/{id}/calendar/getSchedule` (`src/OpenClaw.HostAdapter/SchedulingRoutes.cs`), `POST /users/{assistantMailbox}/sendMail` (`src/OpenClaw.HostAdapter/MailRoutes.cs`), plus the pre-existing `/users/{id}/messages`, `/users/{id}/calendarView`, `/users/{id}/events/{id}` routes. `FreeBusyProjection` (`src/OpenClaw.HostAdapter/FreeBusyProjection.cs`) computes free/busy deterministically from cached events, matching the master's "`calendar/getSchedule` ... computed locally from the cache" mapping.

**Layer 3 — `OpenClaw.Core`** (T1), including `Agent/`. The deterministic engines specified in master Sections 9–10 are implemented as pure, unit-tested C# functions: `TriageEngine`/`TriagePolicy`/`TriageDecision` (5-way decision, Section 9.2), `OwnerPriorityClassifier`/`OwnerSchedulingPolicy` (P0–P4, Section 10.1–10.2), `RecurringMeetingClassifier`/`MovePolicy` (Section 10.3), `SlotProposer`/`WorkingHoursPolicy` (Section 10.4), `SchedulingGate` (the 5-way/priority layering rule), and `MeetingContextNormalizer` (Section 9.2 `normalizeContext`). A `Runtime/` seam (`SchedulingWorker`, `HostAdapterSchedulingService`, `CacheSchedulingCandidateSource`, `SchedulingDtoMapper`) wires these into a `BackgroundService` pipeline. `CalendarPollingWorker`/`MessagePollingWorker` plus `CoreCacheRepository` provide the local polling + SQLite cache + cursor/`ingest_runs` bookkeeping that stands in for Graph subscriptions and `messages/delta` in the Local MVP mapping table.

**Layer 4 — Docker container.** `deploy/docker/openclaw-core.Dockerfile` and `docker-compose.yml` run `OpenClaw.Core` as a locked-down, non-root, read-only container with a health check; `docker-compose.dev.yml` adds a dev container.

**Layer 5 — `openclaw-agent`.** `deploy/docker/openclaw-agent.Dockerfile`, `openclaw-agent-entrypoint.sh`, and `deploy/docker/openclaw-assistant/` (AGENTS.md, SOUL.md, TOOLS.md, `skills/mailbridge_admin/SKILL.md`) implement the agent container referenced in the Local MVP architecture diagram.

**Layer 6 — Install/uninstall.** `scripts/Install.ps1`, `Uninstall.ps1`, `install-mailbridge.ps1`, `uninstall-mailbridge.ps1`, `register-mailbridge-task.ps1`, and the MSIX `installer/` package.

**What is stale in the master document itself.** Master Section "Current state and Local MVP work" (lines 122-128) describes `OpenClaw.HostAdapter` as still exposing only `/v1/*` routes and `OpenClaw.MailBridge` as having six read-only RPC methods with "no outbound or write path." Both statements are superseded by the current code (issues #71–#76 landed). This document's Gap Analysis section below reflects the actual code, not that stale prose.

**What is entirely unstarted.** Stage 1 (cloud/Graph/Exchange RBAC) has zero code: no MSAL/`Azure.Identity` reference, no Key Vault reference, no Bicep/ARM/Terraform IaC, and no Azure Functions/Container Apps project exist anywhere in `src/` (verified by repository-wide grep). `deploy/` contains only Docker artifacts. Stage 2 (calendar-write feature flags, organizer reschedule, attendee propose-new-time) has zero code.

## Automation Feasibility

This section assesses every Stage 1/Stage 2 requirement that touches a third-party system. This repository's CI (`.github/workflows/ci.yml`) runs on `windows-latest` GitHub-hosted runners with no Azure tenant credentials, no Entra app registration, and no Exchange Online connectivity configured (verified: no `Azure`, `Bicep`, `terraform`, `Key Vault`, `MSAL`, or `Azure.Identity` references exist anywhere in `src/`, and the CI workflow has no secrets/credentials steps for those systems). Tenant-dependent verification must therefore be delivered as scripts + runbooks + contract tests against a mocked Graph client, not live calls, until a tenant is provisioned and credentials are supplied out-of-band.

## Risks and Constraints

- **500-line file cap** (`.claude/rules/general-code-change.md`). Several `Core`/`MailBridge` cache files are already close to the cap after the issue #71–#73 field additions (`CacheRepository.cs` at 480/500 lines); any further Graph-field or column addition should default to a new partial-class file rather than growing an existing one.
- **Coverage thresholds** (`.claude/rules/general-unit-test.md`, `csharp.md`): line >= 85%, branch >= 75%, uniform across tiers, with no regression on changed lines. `quality-tiers.yml` at repo root classifies `OpenClaw.Core`/`OpenClaw.HostAdapter` as T1 and `OpenClaw.MailBridge*` as T2/T3, so property-based-test density (>= 1 per pure function) applies to the many pure functions in `OpenClaw.Core.Agent` (`TriageEngine`, `MovePolicy`, `SlotProposer`, etc.) — confirmed already exercised by `tests/OpenClaw.Core.Tests/Agent/*PropertyTests.cs`.
- **Test-framework discrepancy.** `.claude/rules/csharp.md` specifies xUnit + NSubstitute + FluentAssertions as the toolchain, but the actual test suite uses MSTest (`[TestClass]`/`[TestMethod]`, `Microsoft.VisualStudio.TestTools.UnitTesting`) with FluentAssertions (confirmed in `tests/OpenClaw.Core.Tests/Agent/TriageEngineTests.cs:1-12` and consistent across the `tests/` tree). This is a known pre-existing rule-vs-repo mismatch (tracked previously as harness-migration residue, issue #66, per prior research). New feature work should follow the repository's actual MSTest/FluentAssertions convention, not the `csharp.md` text, until the rule file is corrected.
- **No temp files in tests** and **deterministic clock/RNG** (`general-unit-test.md`): `OpenClaw.Core.Agent.SlotProposer` already threads `TimeProvider` through rather than reading `DateTime.UtcNow` directly, matching the `TimeProvider`/`FakeTimeProvider` requirement in `csharp.md`; new Stage 0/1 features (dedupe keys, move-history, audit log) must preserve this pattern.
- **No-COM architecture boundary** (`.claude/rules/architecture-boundaries.md`): all Outlook COM interop is already confined to `OpenClaw.MailBridge` (verified: `Microsoft.Office.Interop.Outlook`/COM member invocation appears only in `OutlookScanner*.cs`, `OutlookComMailSender.cs`, `OutlookComHelpers.cs`, `ComActiveObject.cs`). Any Stage 1 Graph-backed adapter or Azure Functions project must not introduce a COM dependency; `AgentArchitectureBoundaryTests.cs` (`tests/OpenClaw.Core.Tests/Agent/`) already enforces that `OpenClaw.Core.Agent` has no such dependency, and an equivalent architecture test should be added for any new cloud-hosting project.
- **Docker agent boundary.** The `openclaw-agent` container runs `read_only: true`, drops all capabilities, and runs as a non-root user (`docker-compose.yml:60-71`); any new agent-side capability (e.g., a Stage 1 send-on-behalf allowlist file) must be deliverable within that sandbox (bind-mounted config, not writes to the read-only root).
- **Windows-only CI runner.** `.github/workflows/ci.yml` uses `windows-latest` for both `.NET` and PowerShell QC jobs; this is consistent with COM/MSIX requirements but means any new Linux-first Azure Functions tooling should still validate on Windows in CI, or a second CI job should be added rather than assuming cross-platform parity for free.
- **Benchmark-baseline and CI-workflow rules** (`.claude/rules/benchmark-baselines.md`, `ci-workflows.md`) are not currently implicated by this program (no benchmark gate exists yet in `.github/workflows/`), but any future Stage 1 performance-sensitive component (e.g., webhook latency) that adds a benchmark gate must capture baselines on the `windows-latest` runner class with sibling provenance files, and any workflow step that intentionally exercises a failing nested command must reset `$LASTEXITCODE` or `exit 0` explicitly.
